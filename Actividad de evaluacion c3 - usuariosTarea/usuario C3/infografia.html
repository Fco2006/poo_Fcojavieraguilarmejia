01 · Problema

¿Qué resuelve el sistema?

Contexto

Una plataforma digital controla el acceso de distintos tipos de usuarios. Todos comparten información básica, pero cada rol tiene comportamientos y permisos propios.

Admin Cliente Invitado

Sin herencia

  • Código repetido en cada tipo de usuario
  • Cambios básicos requieren editar múltiples clases
  • Difícil de mantener y escalar
  • Sin relación formal entre tipos

Con herencia

  • Clase Usuario base con lo común
  • Clases derivadas extienden sin repetir
  • Polimorfismo: un bucle, distintos comportamientos
  • Fácil agregar nuevos roles

Funcionalidades

  • Validación de email automática
  • Menú interactivo en consola
  • Lista dinámica de usuarios
  • Polimorfismo con recorrido de lista

04 · Conceptos clave

Herencia, sobreescritura y super()

🧬

Herencia

Una clase hija adquiere todos los atributos y métodos de la clase padre. Se especifica poniendo la clase base entre paréntesis.

class Admin(Usuario):
# hereda todo de Usuario
...
✏️

Sobreescritura

La clase hija redefine un método del padre con el mismo nombre para cambiar su comportamiento. Cada tipo de usuario tiene su propio acceso.

def acceso_sistema(self):
# comportamiento propio
print("Admin acceso total")
⚡

super()

Llama al constructor o método del padre desde la clase hija. Evita duplicar código y respeta la cadena de inicialización.

def __init__(self, nombre, email, nivel):
super().__init__(nombre, email)
self.nivel = nivel
05 · Preguntas del entregable

Respuestas

P1

¿Qué ventajas ofrece la herencia frente a repetir código?

La herencia elimina la duplicación: atributos comunes como nombre y email se definen una vez en Usuario. Un solo cambio se propaga a todas las clases hijas, reduciendo errores y facilitando el mantenimiento.

P2

¿Qué métodos fueron heredados?

mostrar_datos(), saludar() y _validar_email() son heredados por las tres clases derivadas. acceso_sistema() también se hereda pero luego es sobrescrito en cada clase hija.

P3

¿Por qué sobrescribir acceso_sistema()?

Porque cada tipo de usuario tiene permisos distintos: Admin gestiona el sistema completo, Cliente accede a compras e historial, e Invitado solo tiene lectura. El override permite que cada clase defina su comportamiento sin alterar a las demás.

P4

¿Diferencia entre atributo heredado y propio?

Un atributo heredado (nombre, email) viene del padre y está disponible en todas las hijas sin redefinirlo. Un atributo propio (nivel_acceso en Admin, puntos en Cliente) es exclusivo de esa clase y se inicializa después de super().__init__().

P5

¿Qué pasaría si todas las clases estuvieran separadas sin heredar?

Habría duplicación masiva: cada clase repetiría nombre, email, validación, mostrar_datos() y saludar(). No habría relación formal, el polimorfismo sería imposible, y cualquier corrección habría que aplicarla en múltiples lugares, aumentando el riesgo de inconsistencias.

06 · Conclusiones

Reflexión final

🧠

¿Qué aprendí?

Aprendí a diseñar sistemas orientados a objetos en Python aplicando herencia real. Entendí la diferencia entre heredar, sobrescribir y reutilizar métodos, y cómo super() conecta la clase hija con la padre. También practiqué polimorfismo para tratar objetos distintos de forma uniforme.

⚡

¿Qué fue lo más difícil?

Lo más retador fue entender cuándo sobrescribir un método completamente y cuándo reutilizar el del padre con super(). También fue interesante implementar la validación de email con expresiones regulares dentro del constructor.

🌍

Aplicaciones reales

Este patrón se usa en sistemas de autenticación web (roles y permisos), plataformas de e-learning con distintos tipos de usuarios, sistemas bancarios con tipos de cuentas, y juegos con clases de personajes que comparten estadísticas base.

🚀

Valor de la POO

La POO no es solo una forma de organizar código, sino una manera de modelar la realidad. Cada clase representa un concepto con sus propias responsabilidades, haciendo el software más comprensible, mantenible y preparado para crecer.
